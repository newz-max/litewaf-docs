# 攻击防护

攻击防护聚合 SQL 注入、XSS、RCE 和路径穿越等托管规则组。它管理托管攻击规则的启停、动作和优先级，不包含访问控制、上传防护、Bot 验证或动态防护能力。

## 适用场景

- 阻断常见 SQL 注入、脚本注入、命令执行和路径穿越尝试。
- 对高误报规则先观察，再逐步切换为阻断。
- 统一管理默认规则集中的托管攻击组。

## 后台入口

进入后台“攻击防护”。按攻击类型或规则组查看托管规则，调整启用状态、动作和优先级。

## 推荐配置

| 场景 | 建议 |
| --- | --- |
| SQL 注入 | 先启用观察，确认业务查询参数无误伤后切换阻断 |
| XSS | 对评论、搜索、富文本入口先观察，避免误伤合法 HTML |
| RCE | 高风险接口建议直接阻断，并保留发布回滚点 |
| 路径穿越 | 对静态资源、下载、文件读取接口重点开启 |

## 匹配范围

攻击防护按托管规则目标字段匹配，例如 URI、Query、Header、Body 摘要或归一化路径。请求体检测需要策略显式开启，并受 Content-Type、路径前缀和大小上限约束。

## 发布后生效

攻击防护组的动作、启停和优先级保存后必须发布新版本。Gateway 使用发布后的本地规则执行，不在请求热路径读取控制面。

## 验证方式

1. 发布观察模式规则，发送包含测试 payload 的请求。
2. 在攻击日志确认 `module=attack-protection` 事件。
3. 切换阻断动作并再次发布，确认命中请求被阻断。

## 日志排查字段

按 `module=attack-protection` 查询。重点查看 `attack_type`、`group_name`、`rule_id`、`rule_name`、`target`、`action`、`score`、`summary`、`disposition`。

## 误伤和风险

- 不要把攻击防护当作访问控制使用；路径允许或阻断应使用访问控制。
- Body 检测会增加处理成本，只对需要的路径开启。
- 日志只保留有限摘要，不保存完整敏感 payload。
